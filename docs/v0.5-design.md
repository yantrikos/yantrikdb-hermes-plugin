# v0.5 design — make memory active

**Status**: draft v2 (replaces v1 input-only scope) · **Target**: yantrikdb-hermes-plugin v0.5.0

The thesis: v0.4.x made the substrate **richer**; v0.5 makes the substrate **active**. Today the agent has to remember memory exists and call the tools; in v0.5 memory shapes every turn whether the agent thinks to ask or not.

## What changes from v1 of this doc

v1 was input-side only (auto-extraction + UI). That brings us to parity with mem0/Letta. v2 keeps those and adds the output-side moves — auto-recall, skill auto-attach, pre-emit contradiction warning, smarter compression, time-aware queries, cross-agent shared brain. The output-side moves are the wedge no other Hermes memory plugin has.

---

## Shipping plan (5 waves, ordered by leverage)

### Wave A — Output side: memory acts before the agent asks

**A1. Auto-recall injection.** Plugin already runs `queue_prefetch()` on each user turn (existing). Today the result is only returned IF the agent calls `_recall`. Change: when the prefetched result has confidence ≥ threshold, inject it into `system_prompt_block()` as `## Relevant memory` for the next turn. Zero new tool calls, zero new latency (prefetch runs in background already). Config gate `YANTRIKDB_AUTO_RECALL=true` (default on).

**A2. Skill auto-attach.** Same shape for skills. `queue_prefetch` learns to also run `skill_search` against the user message; if a skill matches with similarity ≥ 0.7, the skill body lands in the prompt automatically. The agent doesn't need to know `skill_search` exists — the right procedure just appears when relevant. This is where "agent that grows with you" actually happens.

**A3. Pre-emit contradiction warning.** The trickiest. Requires verifying whether Hermes exposes an intercept hook for "before assistant message goes to the user." If yes: when the agent's draft asserts a fact X, plugin pre-checks substrate for ¬X and injects a note. If no hook exists: degrade to **post-emit** via `on_memory_write` of the assistant message — surface contradictions as a follow-up system message in the next turn ("Note: prior session committed to Y; last response said ¬Y"). Generalization of the v0.4.16 silent-failure fix from tool calls to free-form text.

### Wave B — Input side: substrate gets richer automatically

**B1. Auto-extraction `cheap` tier.** Regex + NER on user turns (and on assistant turns the user explicitly confirms — never bare assistant output, per HANDOFF §10.1). Candidates land with `certainty ≤ 0.4`, `source=extracted`, hidden from default recall until `think()` promotes or contradicts them.

**B2. Effectiveness ledger.** `extraction_outcome` records track promote/forget per extractor pattern. New tool `yantrikdb_extraction_stats` exposes precision/recall so noisy patterns can be tuned or disabled.

**B3. `embedding` and `llm` tiers — DEFERRED to v0.5.1.** Cheap-tier validation gates whether the more expensive tiers ever ship.

### Wave C — Visibility: substrate is inspectable in 30 seconds

**C1. Bundled UI.** `yantrikdb-hermes ui [--port 8767] [--open]` starts a single-process stdlib HTTP server. One page, three sections: constellation, recent skills (with outcome ledger), unresolved conflicts. Read-only. NOT a replacement for wysie's dashboard — just the *first-10-minutes-after-install* tool. ~250 LOC total, no new dependencies.

**C2. New tool `yantrikdb_observability`.** Single dispatch returning recall hit-rate, extraction precision, top-N skills by use, namespace breakdown, breaker state. Lets the agent answer "how is my memory doing?" without 6 tool calls.

### Wave D — Smarter under pressure

**D1. `on_pre_compress` hook actually does work.** Today it's a no-op. Change: when Hermes is about to compress context, plugin (a) snapshots the gist of the about-to-be-dropped turns as a high-importance memory, (b) tags them `pre_compression=true` so post-compression recall can resurface them, (c) prefetches based on the *future* prompt shape (the prefix that will survive) so the next recall already has warm results.

**D2. Time-aware queries.** Engine already supports `created_at` filtering; plugin doesn't expose it. Add `since` / `until` / relative ("yesterday", "last week") parameters to `yantrikdb_recall`. The natural-language parsing is local stdlib (`dateparser`); no LLM call.

### Wave E — Shared brain (cross-agent)

**E1. Opt-in cross-agent namespace.** A user's coding agent learns "Pranab prefers tabs"; their WhatsApp agent automatically knows. Today multiple Hermes agents on one machine each have their own namespace. Change: optional `YANTRIKDB_SHARED_BRAIN_NAMESPACE` env var. When set, writes go to both the agent's local namespace AND the shared brain; recalls union both. Conflicts resolved by `think()` as usual.

**Risk**: concurrent writes from multiple agents need conflict semantics. Mitigated by: shared writes carry `source=agent:<name>`; canonicalization can dedupe across agent sources; opt-in default off so existing single-agent users see no behavior change.

## The §10.1 constraint, restated

`sync_turn()` v1 deliberately stores only USER messages because "storing LLM output as fact amplifies hallucination." Every v0.5 feature respects this:

- B1 extraction: only from user turns or explicitly-user-confirmed assistant assertions
- A3 contradiction warning: reads from substrate, doesn't write LLM output as fact
- D1 compression snapshot: writes a high-confidence summary tagged `source=compression_summary` (provenance, not raw assertion) — recall surfaces it with that tag visible
- E1 shared brain: same writing rules apply across agents; LLM output never written without user-confirmation gate

## Risks & mitigations

1. **Prompt budget bloat from auto-injection.** A1 + A2 + D1 all add to `system_prompt_block`. Mitigation: each section bounded (3 recall hits, 2 skill bodies, 1 compression summary), total injection budget capped at ~600 tokens, configurable. If the user has `YANTRIKDB_PROMPT_BUDGET=0` we disable auto-inject entirely.

2. **A3 needs hook discovery.** If Hermes lacks a pre-emit intercept, A3 degrades to post-emit follow-up. Acceptable but less elegant. Decision deferred to milestone-1 spike.

3. **Extraction false-positives still real.** Mitigated as in v1 design: candidate-state + `think()` gate + effectiveness ledger. If real-use precision < 70%, ship with `B1.enabled=false` default until tuned.

4. **Scope balloon.** Five waves is a lot. Mitigation: each wave ships independently behind feature flags. v0.5.0 ships Wave A only as MVP; B/C/D/E land as 0.5.x point releases over 2-4 weeks. We commit to the architecture; we ship in slices.

5. **Cross-agent semantics under concurrency.** E1's "write to two namespaces" needs careful think() behavior. Mitigated by opt-in default off + first ships with single-writer semantics (warn on concurrent shared-brain writers).

## What's NOT in v0.5

- `yantrikdb-migrate` from mem0/Letta/Zep — separate package, separate cadence
- CrewAI / BeeAI adapters — separate packages
- Multi-modal (image/audio) — engine doesn't support, deferred to v0.6+
- LLM-tier extraction — deferred to v0.5.1 pending cheap-tier validation
- Authenticated / multi-user UI — wysie's dashboard covers that surface
